DevRev unifies product development (Build), support (Support), and CRM (Grow) around one knowledge graph. Use the `mcp__devrev__*` tools to operate on it; fall back to the `devrev` CLI (via bash) only where noted below.

Spell the company/platform name `DevRev` (or lowercase `devrev` in identifiers, paths, and commands) ‚Äî never `Devrev`. The only exception is autogenerated code, where you preserve whatever the generator emits.

Prefer the existing conversation context. If the user already gave you the work item text or a current-task summary, use it directly ‚Äî don't redraft it.

If the target object or the requested operation is unclear, ask one concise clarifying question instead of guessing.

## Most useful tools

- **`mcp__devrev__get_tool_metadata`** ‚Äî call FIRST in a fresh DevRev session for ID-resolution strategy and workflow guidance.
- **`mcp__devrev__fetch_object_context`** ‚Äî the workhorse for reading. Accepts display IDs (`ISS-123`, `TKT-456`) and full DON IDs interchangeably. Use before updating/linking to understand current state.
- **`mcp__devrev__hybrid_search`** ‚Äî discover object IDs by name; search a `namespace` (`issue`, `ticket`, `enhancement`, `article`, `feature`, `capability`, `product`, `vista`, ‚Ä¶). Use this to find parts/enhancements.
- **`mcp__devrev__discover_schema`** ‚Äî call with `action_name='create_<type>'` / `update_<type>' / `list_<type>'` BEFORE create/update/list to learn required fields, enums, and constraints.
- **`mcp__devrev__create_object` / `update_object` / `list_objects`** ‚Äî mutate/list work items (pass the `action_name` from discover_schema).
- **`mcp__devrev__link_objects`** ‚Äî link items (e.g. `link_ticket_with_issue`); `link_issue_with_issue` supports `link_type` (`is_parent_of`, `is_dependent_on`, `is_duplicate_of`).
- **`mcp__devrev__get_valid_stage_transitions`** ‚Äî check allowed stage moves before transitioning.

These `mcp__devrev__*` tools are already available whenever this skill is loaded ‚Äî call them directly. Do not run `ToolSearch` to rediscover them. `get_tool_metadata` and `discover_schema` results are stable for the session; fetch each at most once and reuse the result rather than re-querying before every call.

## Resolving an org slug ‚Üí dev org ID

When the user gives you an **org slug** and you need the **dev org ID** (dev_oid), resolve it via the internal `rev-orgs.list` API, filtering by `display_name` (the slug is the org's display name). The slug is the first path segment of an org URL ‚Äî `https://app.devrev.ai/<slug>` (e.g. `app.devrev.ai/devrev` ‚Üí slug `devrev`, `app.devrev.ai/maplestudio-playground-30-134320` ‚Üí slug `maplestudio-playground-30-134320`).

```bash
curl -s -X POST "https://api.devrev.ai/internal/rev-orgs.list" \
  -H "Authorization: $DEVREV_API_KEY" -H "Content-Type: application/json" \
  -d '{"display_name":["<slug>"],"limit":5}'
```

Notes:
- Auth header is the **raw `$DEVREV_API_KEY`** ‚Äî NO `Bearer` prefix. Filter param is `display_name` (an array); `q`/`search` return HTTP 400.
- From the matching rev_org, read **`external_ref`** (issuer `devrev:platform`) ‚Äî that is the **dev org DON**, e.g. `don:identity:dvrv-us-1:devo/1g5jplBimm`. The bare ID is the `devo/<id>` suffix (`1g5jplBimm`), and the `DEV-` form is `DEV-1g5jplBimm`.
- `custom_fields.tnt__dev_org_slug` confirms the slug; `id` (`don:...:revo/<id>`, `REV-*`) is the **rev_org** DON ‚Äî a different object, don't confuse it with the dev org.
- **Do not** use `id_v1` (it shows `don:DEV-0:...` for everything): `DEV-0` is the DevRev-internal owner org, not the customer's dev org.
- If `rev-orgs.list` errors or returns nothing, fall back to `search.core` (namespace `rev_org`) to find the exact `display_name`, then retry with that value. If neither resolves it, tell the user plainly what you tried and ask for the dev org ID (`DEV-xxx`) ‚Äî never guess or fabricate one.

## Work URLs

Always render DevRev work URLs in canonical app form:
- `https://app.devrev.ai/devrev/works/ISS-123`
- `https://app.devrev.ai/devrev/issue/ASDAT-26`

Articles (ART-*) live under a different path ‚Äî the knowledge-base settings area, not `works`:
- `https://app.devrev.ai/devrev/settings/knowledge-base/articles/ART-84487`

## Fetching / reading

When you fetch an object, keep the response lean to preserve context. Unless the user asked for a specific subset of fields, return the full object but always:
- remove `shared_with` and `tags`
- remove nested `display_picture` and `thumbnail` from any identity/user-reference fields (`owned_by`, `created_by`, `modified_by`, `authored_by`, and similar) anywhere in the object

Include comments only when the user explicitly asks for them or when they're clearly needed to answer the request.

## Articles (ART-*)

Article **bodies cannot be handled through the MCP.** The body lives in a separate
`devrev/rt` artifact (ProseMirror JSON), not inline on the article object:
- `create_article`'s `content` field is broken server-side (fails with
  `missing_required_field: file_name`).
- `update_article` has **no** body/content field at all ‚Äî only title, status,
  owner, parts, tags, attachments.

So all body reads/writes go through the DevRev **REST API** with `curl` + `$DEVREV_API_KEY`
(raw token, NO `Bearer` prefix). Always `curl` the signed URL directly (or via the
helper below) and parse the raw JSON with `jq`. A helper script wraps the fiddly
artifact-upload dance ‚Äî use it rather than hand-rolling curl for writes:

```
${SKILL_PATH}/scripts/handle-article-body {get|create|upload} ‚Ä¶
```

### The devrev/rt body format

The body artifact is JSON with two top-level keys ‚Äî preserve this shape:
```json
{ "article": { "type": "doc", "content": [ ‚Ä¶ProseMirror nodes‚Ä¶ ] }, "artifactIds": [] }
```
`article` is a ProseMirror doc: nodes like `heading` (with `attrs.level`),
`paragraph`, `bulletList`/`orderedList` ‚Üí `listItem` ‚Üí `paragraph`, and leaf `text`
nodes holding the string (bold/italic via `marks`). `artifactIds` lists artifacts
embedded in the body (images, etc.) ‚Äî usually `[]`. **To change an article, fetch the
existing body first and mutate its `content` tree**, then create a replacement article
from it (bodies can't be updated in place ‚Äî see below) so the structure is retained.

### Reading a body

Never use `extracted_content` ‚Äî that `text/plain` artifact is AI-facing plaintext,
not the real body. The real body is the `resource.artifacts[]` entry whose
`file.name` is `Article` (its `file.type` is `devrev/rt`).

```bash
${SKILL_PATH}/scripts/handle-article-body get ART-84444    # or a full DON / bare numeric id
```
Prints the raw `devrev/rt` JSON. Under the hood: `articles.get` ‚Üí pick the `Article`
artifact ‚Üí `artifacts.locate` for a signed URL ‚Üí download. When the user wants the
body, return the article identifiers plus this body (or a Markdown rendering of it).

If no `Article` artifact exists, the download isn't valid JSON, or it lacks an
`article` key, report the error ‚Äî do not fall back to `extracted_content` silently.

### Creating an article with a body

```bash
${SKILL_PATH}/scripts/handle-article-body create <rt-json-file> "<title>" \
  <owner-devu-id> <applies-to-part-id> [status]     # status defaults to draft
```
`owner` is a `devu` id (`get_self` for yourself); the part id comes from
`hybrid_search`. This uploads the rt file as an artifact and sets the article's
`resource:{artifacts:[‚Ä¶]}`. (Title/owner/part/status alone can also go through the
MCP `create_article` ‚Äî but only the REST path attaches a body.)

### Updating a body ‚Äî NOT POSSIBLE; replace the article instead

**You cannot change an existing article's body.** Pointing an existing article at a
new `devrev/rt` artifact via `articles.update` (any field ‚Äî `artifacts.set`,
`content_artifact`, ‚Ä¶) leaves it un-renderable in the KB: the page 404s as "item
doesn't exist anymore". This was verified exhaustively ‚Äî the resulting article
object, artifact object, and body bytes are byte-for-byte identical to a
freshly-created article that renders fine, so the missing piece is a create-only
side effect (render/search indexing) that update never fires, and no REST/MCP field
reproduces it. There is intentionally no `set` command in the helper.

To change a published article's body, **create a replacement and delete the old one**:
1. `handle-article-body get ART-123 > /tmp/body.json` ‚Äî fetch the current body.
2. Edit `/tmp/body.json` (mutate its ProseMirror `content` tree with jq/an editor).
3. `handle-article-body create /tmp/body.json "<title>" <owner> <part> [status]` ‚Äî create
   the replacement. Replicate the original's other fields (`parent`, `shared_with`,
   `tags`, `description`) via MCP `update_object` (`action_name='update_article'`) on
   the new article if they matter; `scope` is derived from sharing automatically.
4. Verify the new article renders in the KB, then delete the old one (`articles.delete`).

Non-body article fields (title, status, owner, parts, sharing) on an EXISTING article
still work fine via the MCP `update_object` with `action_name='update_article'` ‚Äî only
the body/content is unchangeable.

## Creating objects

Title and description must be the user's exact words.

- Confirm the exact title and exact description before creating (use AskUserQuestion if anything is ambiguous).
- If either is missing, stop and ask ‚Äî never draft your own.
- After confirmation, use the values verbatim: do not rewrite, summarize, improve, expand, or shorten them.

## Current-work issue creation

To create an issue for the **current work / finish workflow** (with the AirSync Data Plane defaults ‚Äî owner, team, sprint, priority, part heuristics, stage ‚Üí `in_development`), use the dedicated **`create-devrev-issue`** skill instead. It runs non-interactively in a forked context and owns those defaults. Do not duplicate that logic here.

This skill handles all other (interactive / general) DevRev creation per "Creating objects" above.

## Moving to in_review

After the pull request for the work has been opened, transition the issue stage from `in_development` to `in_review` ‚Üí `don:core:dvrv-us-1:devo/0:custom_stage/17` (this is a valid `in_development` ‚Üí `in_review` transition; do not attempt a direct `triage` ‚Üí `in_review` move).
If the `in_review` update fails, report it and note the issue remained in `in_development`.
