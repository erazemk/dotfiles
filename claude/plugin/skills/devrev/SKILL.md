DevRev unifies product development (Build), support (Support), and CRM (Grow) around one knowledge graph. Use the `mcp__devrev__*` tools to operate on it; fall back to the `devrev` CLI (via bash) only where noted below.

Spell the company/platform name `DevRev` (or lowercase `devrev` in identifiers, paths, and commands) ŌĆö never `Devrev`. The only exception is autogenerated code, where you preserve whatever the generator emits.

Prefer the existing conversation context. If the user already gave you the work item text or a current-task summary, use it directly ŌĆö don't redraft it.

If the target object or the requested operation is unclear, ask one concise clarifying question instead of guessing.

## Most useful tools

- **`mcp__devrev__get_tool_metadata`** ŌĆö call FIRST in a fresh DevRev session for ID-resolution strategy and workflow guidance.
- **`mcp__devrev__fetch_object_context`** ŌĆö the workhorse for reading. Accepts display IDs (`ISS-123`, `TKT-456`) and full DON IDs interchangeably. Use before updating/linking to understand current state.
- **`mcp__devrev__hybrid_search`** ŌĆö discover object IDs by name; search a `namespace` (`issue`, `ticket`, `enhancement`, `article`, `feature`, `capability`, `product`, `vista`, ŌĆ”). Use this to find parts/enhancements.
- **`mcp__devrev__discover_schema`** ŌĆö call with `action_name='create_<type>'` / `update_<type>' / `list_<type>'` BEFORE create/update/list to learn required fields, enums, and constraints.
- **`mcp__devrev__create_object` / `update_object` / `list_objects`** ŌĆö mutate/list work items (pass the `action_name` from discover_schema).
- **`mcp__devrev__link_objects`** ŌĆö link items (e.g. `link_ticket_with_issue`); `link_issue_with_issue` supports `link_type` (`is_parent_of`, `is_dependent_on`, `is_duplicate_of`).
- **`mcp__devrev__get_valid_stage_transitions`** ŌĆö check allowed stage moves before transitioning.

These `mcp__devrev__*` tools are already available whenever this skill is loaded ŌĆö call them directly. Do not run `ToolSearch` to rediscover them. `get_tool_metadata` and `discover_schema` results are stable for the session; fetch each at most once and reuse the result rather than re-querying before every call.

## Resolving an org slug ŌåÆ dev org ID

When the user gives you an **org slug** and you need the **dev org ID** (dev_oid), resolve it via the internal `rev-orgs.list` API, filtering by `display_name` (the slug is the org's display name). The slug is the first path segment of an org URL ŌĆö `https://app.devrev.ai/<slug>` (e.g. `app.devrev.ai/devrev` ŌåÆ slug `devrev`, `app.devrev.ai/maplestudio-playground-30-134320` ŌåÆ slug `maplestudio-playground-30-134320`).

```bash
curl -s -X POST "https://api.devrev.ai/internal/rev-orgs.list" \
  -H "Authorization: $DEVREV_API_KEY" -H "Content-Type: application/json" \
  -d '{"display_name":["<slug>"],"limit":5}'
```

Notes:
- Auth header is the **raw `$DEVREV_API_KEY`** ŌĆö NO `Bearer` prefix. Filter param is `display_name` (an array); `q`/`search` return HTTP 400.
- From the matching rev_org, read **`external_ref`** (issuer `devrev:platform`) ŌĆö that is the **dev org DON**, e.g. `don:identity:dvrv-us-1:devo/1g5jplBimm`. The bare ID is the `devo/<id>` suffix (`1g5jplBimm`), and the `DEV-` form is `DEV-1g5jplBimm`.
- `custom_fields.tnt__dev_org_slug` confirms the slug; `id` (`don:...:revo/<id>`, `REV-*`) is the **rev_org** DON ŌĆö a different object, don't confuse it with the dev org.
- **Do not** use `id_v1` (it shows `don:DEV-0:...` for everything): `DEV-0` is the DevRev-internal owner org, not the customer's dev org.
- If `rev-orgs.list` errors or returns nothing, fall back to `search.core` (namespace `rev_org`) to find the exact `display_name`, then retry with that value. If neither resolves it, tell the user plainly what you tried and ask for the dev org ID (`DEV-xxx`) ŌĆö never guess or fabricate one.

## Fetching / reading

When you fetch an object, keep the response lean to preserve context. Unless the user asked for a specific subset of fields, return the full object but always:
- remove `shared_with` and `tags`
- remove nested `display_picture` and `thumbnail` from any identity/user-reference fields (`owned_by`, `created_by`, `modified_by`, `authored_by`, and similar) anywhere in the object

Include comments only when the user explicitly asks for them or when they're clearly needed to answer the request.

## Article fetches (ART-*)

Articles need a CLI + webfetch round-trip ŌĆö the MCP fetch alone won't give you the body:

1. Derive the numeric ID: the suffix after `ART-`, or the last numeric path segment if given a DON.
2. Run via bash: `devrev -o devrev -e prod articles show don:core:dvrv-us-1:devo/0:article/<id> | jq` ŌĆö treat this CLI JSON as the source of truth.
3. In the CLI JSON, inspect `article.resource.artifacts`; find the artifact whose `file.name` is `Article`. (`file.type` is informational ŌĆö it may be `text/markdown`, `devrev/rt`, etc.; don't require a specific value.)
4. `webfetch` that artifact's `original_url` and parse the response as JSON.
5. Apply the normal field-stripping rules to the article object, then add a `body` field set to the fetched body's `article` field.
6. If no matching artifact with `original_url` exists, the fetched response isn't valid JSON, or it lacks an `article` field, report the error ŌĆö do not fall back silently.

When the user asks only for an article's body/content, return the article identifiers plus the `body` field.

## Creating objects

Title and description must be the user's exact words.

- Confirm the exact title and exact description before creating (use AskUserQuestion if anything is ambiguous).
- If either is missing, stop and ask ŌĆö never draft your own.
- After confirmation, use the values verbatim: do not rewrite, summarize, improve, expand, or shorten them.

## Current-work issue creation

To create an issue for the **current work / finish workflow** (with the AirSync Data Plane defaults ŌĆö owner, team, sprint, priority, part heuristics, stage ŌåÆ `in_development`), use the dedicated **`create-devrev-issue`** skill instead. It runs non-interactively in a forked context and owns those defaults. Do not duplicate that logic here.

This skill handles all other (interactive / general) DevRev creation per "Creating objects" above.

## Moving to in_review

After the pull request for the work has been opened, transition the issue stage from `in_development` to `in_review` ŌåÆ `don:core:dvrv-us-1:devo/0:custom_stage/17` (this is a valid `in_development` ŌåÆ `in_review` transition; do not attempt a direct `triage` ŌåÆ `in_review` move).
If the `in_review` update fails, report it and note the issue remained in `in_development`.
